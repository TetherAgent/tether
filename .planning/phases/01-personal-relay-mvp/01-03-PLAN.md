<objective>
Add a Gateway-side outbound relay client that registers existing sessions with the Relay, serves replay/live events from `session_events`, and forwards remote input/resize to existing Gateway-owned PTY sessions without changing direct LAN/localhost Gateway behavior.
</objective>

<threat_model>
## Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| Relay-routed writes bypass existing observe/control rules | high | Relay client must treat `mode: 'observe'` as read-only and only process input/resize for control clients. |
| Relay creates new processes remotely | high | Relay client only handles list/subscribe/input/resize/detach frames for existing session IDs; no provider command frame handling. |
| Direct Gateway behavior regresses | high | Relay client is optional and starts only when relay URL and secret are configured. Existing `/api` routes remain unchanged. |
| Output leaks unmasked secrets | medium | Relay forwards events already produced by `PtySessionManager`/Store, where terminal output and stored user input are masked. |
</threat_model>

  <action>
    Create `apps/gateway/src/relay-client.ts` exporting `RelayClientOptions`, `RunningRelayClient`, and `startRelayClient(options)`.
    `RelayClientOptions` must include `url`, `secret`, `gatewayId`, `store`, and optional `ptySessions`.
    Use `ws` as an outbound WebSocket client to connect to the relay `/gateway` endpoint.
    On open, send a `gateway.auth` frame with `secret` and `gatewayId`, followed by a `gateway.sessions` frame containing `store.listSessions()` mapped to protocol `RelaySession`.
    Handle `client.list` by sending current `gateway.sessions`.
    Handle `client.subscribe` by replaying `store.listEvents(sessionId, after, 5000)` as `gateway.replay`, sending a replay completion represented by the protocol, and subscribing to `ptySessions.subscribe(sessionId, ...)` for live `gateway.event` forwarding.
    Handle `client.input` by calling `ptySessions.write(sessionId, { clientId, data })` only when mode is not `observe`.
    Handle `client.resize` by calling `ptySessions.resize(sessionId, clientId, cols, rows)` only when mode is not `observe`.
    Handle `client.detach` by removing any relay subscription for that client/session.
    Implement reconnect with a bounded retry delay of 1000ms to 5000ms while `close()` has not been called.
  </action>
