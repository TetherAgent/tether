## Summary

Phase 14 touches five distinct systems: the MySQL `gateways` table schema, the CLI gateway login flow, the server-side gateway-auth controller and service, the Relay routing logic, and the web chat UI. The work is largely additive except for two high-risk destructive changes: simplification of `auth.json` (which breaks four call sites across two packages) and removal of relay fallbacks (which changes behavior for clients that omit `gatewayId`).

**Critical finding:** `007_remove_workspace.sql` already dropped `workspace_id` from the `gateways` table and dropped `fk_gateways_workspace`. The CONTEXT.md description of D-03b ("drop FK, make nullable") does not match the current schema state. The migration for Phase 14 must NOT attempt to drop `fk_gateways_workspace` or touch `workspace_id`.

**Primary recommendation:** Implement in strict order: (1) DB migration, (2) server-side service/controller changes, (3) CLI auth.json simplification + daemon.ts refactor, (4) relay routing changes, (5) web UI. The relay and daemon changes are mutually dependent — both must ship together or the gateway cannot connect.

### 8. JWT decode capability — CLI (`apps/cli/src/main.ts`)

`decodeTokenPayload` function at line 1490-1498 already exists in the CLI:

```typescript
function decodeTokenPayload(token: string): { expiresAt?: unknown } | undefined {
  const parts = token.split('.');
  if (parts.length !== 3) return undefined;
  try {
    return JSON.parse(Buffer.from(parts[1], 'base64url').toString('utf8')) as { expiresAt?: unknown };
  } catch {
    return undefined;
  }
}
```

The CLI uses this to extract `expiresAt` from the access token during `performGatewayLogin`. No external JWT library — raw base64url decode. The gateway package has **no JWT library** in its dependencies (only `@hono/node-server`, `hono`, `node-pty`, `ws`). A similar `decodeTokenPayload` function must be added to `apps/gateway/src/daemon.ts` and/or `relay-client.ts`.

## Breaking Changes

Ordered by risk (highest first).

### BC-1: auth.json workspace_id already dropped — D-03b is WRONG

**Risk: HIGH — Plan will fail if followed literally.**

`007_remove_workspace.sql` already executed `ALTER TABLE gateways DROP FOREIGN KEY fk_gateways_workspace` and `ALTER TABLE gateways DROP COLUMN workspace_id`. The CONTEXT.md D-03b says "drop FK `fk_gateways_workspace`, make `workspace_id` nullable" — but both are already gone.

**Action:** Phase 14 migration must NOT touch `workspace_id` or `fk_gateways_workspace`. They do not exist. The migration is only: add columns, drop old unique key, add new unique key.

### BC-2: auth.json simplification breaks 4 call sites across 2 packages

**Risk: HIGH — Gateway goes dark if any one call site is missed.**

Files and locations that validate or use `gatewayId`/`accountId` from auth.json:

| File | Lines | What breaks |
|------|-------|-------------|
| `apps/gateway/src/daemon.ts` | 1136-1153 (`parseGatewayAuthState`) | Parses `gatewayId` and `accountId` — returns invalid if fields absent |
| `apps/gateway/src/daemon.ts` | 188, 687, 1083 | Uses `authState.value.gatewayId` at runtime |
| `apps/gateway/src/relay-client.ts` | 1059-1091 (`resolveRelayAuth`) | Validates `typeof parsed.gatewayId !== 'string'` — returns undefined if absent |
| `apps/cli/src/main.ts` | 1466-1483 (`readGatewayAuthState`) | Validates `gatewayId` and `accountId` |
| `apps/cli/src/main.ts` | 1369-1376 (`performGatewayLogin`) | Writes `gatewayId` and `accountId` to auth.json |
| `apps/cli/src/main.ts` | 140-148 (`GatewayAuthState` type) | Type includes `gatewayId` and `accountId` |

The gateway package has **no JWT library**. It must add a `decodeTokenPayload` helper (same pattern as CLI line 1490-1498: raw base64url split). The decoded payload structure from the access token must include `gatewayId`, `accountId`, and `userId` — confirmed because `issueGatewayTokenBundle` in `apps/server/app/service/auth.ts` embeds these in the JWT claims.

### BC-3: Relay fallback removal — D-13 changes `client.chat/list-providers/cwd-suggest` behavior

**Risk: HIGH — All existing web clients break until web UI also ships `gatewayId`.**

Currently `ensureClientGatewayId` (relay.ts line 738-750) falls back to `firstGatewayForScope`, so single-gateway users work without `gatewayId` in frames. After removal, `forwardToGateway(undefined, ...)` is called, which currently sends `gateway_unavailable` (not `gateway_required`). A new check must be added before `forwardToGateway` calls in the `client.chat`, `client.list-providers`, `client.cwd-suggest` cases: if `frame.gatewayId` is absent, return `gateway_required` immediately without calling `forwardToGateway`.

The relay.ts line 501 fallback (during client auth) is a separate issue: removing it means `client.gatewayId` is `undefined` at connection time. The web sends no `gatewayId` in the auth frame currently. This is fine because Phase 14 removes the WS-level binding and moves `gatewayId` into each frame.

### BC-4: `GatewayRecord.status` type gap

**Risk: MEDIUM — TypeScript will reject `'revoked'` if type is not updated.**

`runtime.ts` line 51: `status: 'online' | 'offline'`. Must add `| 'revoked'`. Also affects `gatewayFromRow` in `gatewayRepository.ts` line 24 which has `row.status === 'offline' ? 'offline' : 'online'` — this will map `'revoked'` to `'online'`. Must be updated.

### BC-5: `saveGateway` SQL uses wrong unique key for upsert

**Risk: MEDIUM — After migration, upsert by `account_id, user_id` no longer exists as a constraint.**

`gatewayRepository.ts` line 42: `ON DUPLICATE KEY UPDATE` relies on the current unique key `uq_gateways_account_user`. After migration, that key is dropped. The SQL must be rewritten to use `uq_gateways_device_key (account_id, user_id, device_key)`. The upsert semantics also change: D-08 says "do NOT update `name`" on conflict. Current code updates `name` (line 45).

### BC-6: CLI `waitForGatewayAuthCallback` reads `gatewayId` and `accountId` from callback

**Risk: MEDIUM — If server stops sending these, CLI login silently fails.**

Current callback URL (set by `gateway-auth-page.tsx` line 56-58) includes `gatewayId` and `accountId`. The CLI reads them at lines 1417-1422. Per D-09, the CLI will decode `gatewayId` and `accountId` from the access token JWT instead of storing them in auth.json. However, the CLI still needs to get them from somewhere during login. Possible approaches:
1. Server still returns them in the callback (easiest, CLI decodes from token for storage)
2. CLI decodes them from the `gatewayAccessToken` JWT (zero server change)

Option 2 is cleanest: the callback already includes `gatewayAccessToken` containing `gatewayId`/`accountId` in its payload. CLI can decode them from the token at line 1365 where it already calls `decodeTokenPayload`.

## Reusable Patterns

### JWT decode (no library needed)

Follows the pattern already in `apps/cli/src/main.ts` line 1490-1498:
```typescript
function decodeTokenPayload(token: string): Record<string, unknown> | undefined {
  const parts = token.split('.');
  if (parts.length !== 3) return undefined;
  try {
    return JSON.parse(Buffer.from(parts[1]!, 'base64url').toString('utf8'));
  } catch { return undefined; }
}
```
Copy this into `apps/gateway/src/daemon.ts`. No new dependency needed.

### `gatewayAuthPath()` — shared path logic

Both `apps/gateway/src/daemon.ts` (line 1132-1134) and `apps/cli/src/main.ts` (line 1334) independently implement the same function. No shared package — both read `TETHER_AUTH_PATH ?? ~/.tether/auth.json`. Phase 14's `device.json` should use the same `~/.tether/` dir: `TETHER_DEVICE_PATH ?? path.join(os.homedir(), '.tether', 'device.json')`.

### Repository upsert pattern

`gatewayRepository.saveGateway` uses MySQL `INSERT ... ON DUPLICATE KEY UPDATE`. This pattern is correct for upsert. The new `saveGateway` / `upsertGatewayByDeviceKey` must keep this pattern but use the new unique key.

### Relay account-scoped broadcast pattern

`broadcastGatewayUnavailableForScope` (relay.ts line 785-792) and `broadcastGatewayStatus` (line 794-803) both filter by `clientCanUseGateway(client.scope, gatewayScope)`. Same pattern must apply to any new broadcast involving the gateway list.

### Multi-tenant isolation (CLAUDE.md R1-R4)

- R1: Use `firstGatewayForScope(accountId)` not `firstConnectedGateway()` — applicable to D-12.
- R2: Session storage carries `gatewayId` — already done in relay.ts line 263.
- R3: Broadcasts filter by accountId — applicable to any new gateway list push.
- R4: New routing paths need multi-account isolation tests.

### Egg server layer

- New route: `GET /api/server/gateways` needs `requireNormalAccess` middleware, controller reads `ctx.state.auth.accountId` and `ctx.state.auth.userId`, calls service, returns `ctx.success(data)`.

## Implementation Notes

### Note 1: Migration numbering

Existing SQL files: `001` through `007`. Next is `008`. Suggest `008_multi_device_gateway.sql`.

### Note 2: D-03b is already done — DO NOT re-do

The FK `fk_gateways_workspace` and column `workspace_id` were removed by `007_remove_workspace.sql`. The new migration must NOT include any attempt to drop them. The `001_init.sql` still shows `workspace_id NOT NULL` because the init file reflects the original schema, not the current applied state.

### Note 3: `gateway.ts` `bindGatewayForUser` must not update `name` on upsert conflict

Current line 105: `gateway.name = input.gatewayName ?? gateway.name ?? 'local-gateway'` overwrites name unconditionally. D-08 says: on UPDATE, do not touch `name`. On INSERT (new device), default `name` to `hostname`. This logic lives in the service before calling `saveGateway`.

### Note 4: relay.ts `ensureClientGatewayId` — what to do with it after D-13

After removing the `firstGatewayForScope` fallback (lines 746-747), `ensureClientGatewayId` becomes:
```typescript
function ensureClientGatewayId(clientId: string): string | undefined {
  return clients.get(clientId)?.gatewayId ?? clients.get(clientId)?.scope?.gatewayId;
}
```
But since D-10 moves `gatewayId` into each frame, this function's callers (`client.chat`, `client.list-providers`, `client.cwd-suggest`) should no longer call `ensureClientGatewayId` at all — they should read `frame.gatewayId` directly. The function may become dead code after Phase 14.

### Note 5: Web `relayGatewayId` state and multi-gateway

Currently `relayGatewayId` tracks the single connected gateway from the `hello` and `gateway.status` frames. For multi-device support, the web needs:
1. A list of gateways fetched from `GET /api/server/gateways`
2. A `selectedGatewayId` state (the user's active selection)
3. Online status overlay from relay `gateway.status` frames (already pushed per-gateway)

The `relayGatewayId` state can be repurposed or renamed as `selectedGatewayId`. The `hello` frame's `gatewayId` field becomes less meaningful (Phase 14 removes the WS-level binding), but it's still sent by relay.

### Note 6: `gateway_required` vs `gateway_unavailable`

Two distinct error codes are needed:
- `gateway_unavailable` (existing): Gateway is known but offline/disconnected
- `gateway_required` (new, D-11): Frame arrived without a `gatewayId` field at all

The web error handler at `chat-panel.tsx` line 792-811 currently handles `gateway_unavailable`. It must also handle `gateway_required` (show selector) and `gateway_unauthorized` (show error).

### Note 7: `gateway_refresh_tokens` table has `workspace_id`

`001_init.sql` shows `gateway_refresh_tokens` with `workspace_id`. `007_remove_workspace.sql` drops it. This table's rows are not affected by Phase 14 (no workspace column in the applied schema).

### Note 8: The `device.json` does not exist yet anywhere

No code in the codebase reads or writes `~/.tether/device.json`. Phase 14 must add this to `apps/cli/src/main.ts` in `performGatewayLogin`: read device.json, generate `deviceKey` if absent, write back, then include `deviceKey` in the browser URL.

### Note 9: `validateAccessToken` in daemon is a network call, not JWT decode

`validateAccessToken` (daemon.ts line 1182) makes an HTTP call to `POST /api/server/token/validate`. For `getGatewayIdentity`, we do NOT want a network call — we want to decode the JWT locally (no signature verification needed, just payload extraction). This is safe because the fields extracted (`gatewayId`, `accountId`, `userId`) are only used internally to identify the gateway; the token itself is still validated server-side when the gateway connects to Relay.

### Note 10: `workspaceId` in live `auth.json`

The live `~/.tether/auth.json` on this machine has `workspaceId` (legacy field from before 007 migration). After Phase 14, parsing must tolerate extra unknown fields. Since `parseGatewayAuthState` uses `JSON.parse(raw) as Partial<GatewayAuthState>` and only checks the fields it cares about, extra fields are already tolerated.
