<objective>
Implement the remote auth runtime inside `apps/server`: first-owner registration at `/register`, normal and management login/refresh/logout, Gateway bind/refresh, notification WebSocket, revoke handling, and identity-bearing audit writes. This is the server-side heart of Phase 5.
</objective>

<threat_model>
## Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| Normal and management identity realms are mixed and a management token can later leak into session control paths | high | Separate controllers/services/tables/token classes for normal vs management auth and test the wrong-class cases. |
| Revoked tokens still refresh or continue operating because `jti` blocklist is incomplete | high | Use Redis-backed revoke checks for refresh/logout/bind flows and make tests cover revoked `jti` behavior. |
| Notification socket leaks account/session metadata across accounts | high | Notification auth middleware must join rooms using account/user identity from validated tokens only and emit metadata-only events. |
</threat_model>

## Must Haves

### Truths
- `/register` is the required first-owner path; `/setup?token=...` must not come back as the main Phase 5 flow.
- Normal and management tokens are separate and non-interchangeable.
- Notification WS is metadata-only; PTY bytes stay on Gateway/Relay session sockets.

### Checks
- Server exposes concrete auth, bind, revoke, and notification endpoints.
- Audit rows contain identity fields without raw secrets.
- Revoke checks are wired into token middleware/service flows.
</must_haves>
