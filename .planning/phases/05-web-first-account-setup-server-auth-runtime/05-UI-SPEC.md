# Phase 5 — UI Design Contract

> Visual and interaction contract for Web-first Account Setup & Server Auth Runtime.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Scope:** This contract covers only the Phase 5 Web UI surfaces: `/register`, `/login`,
> `/admin/register`, `/admin/login`, and the authenticated session shell updates needed to
> support token-gated access. Full management console pages (users, devices, Gateways,
> audit, login analytics) are Phase 6 and out of scope here.

## Typography

Phase 5 auth pages use the existing system font stack. shadcn generates CSS variables
(`--font-sans`) that will pick up the same stack. No custom web font is introduced.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px | 400 (regular) | 1.5 | Form helper text, metadata, error messages |
| Label | 14px | 400 (regular) | 1.4 | Form field labels, card secondary text |
| Heading | 20px | 650 (semibold) | 1.2 | Auth card page title ("Sign in", "Create account") |
| Display | 15px | 650 (semibold) | 1.2 | App header h1 (existing, unchanged) |

**Weight constraint:** Use exactly two weights in Phase 5 auth page components:
400 (regular) for body, helper text, and labels; 650 (semibold) for headings and CTAs.
shadcn's internal label component may render at a slightly different weight as a
browser/framework default — that is acceptable as a pass-through, but weight 500 is not
a declared weight in this contract.

## Color

The existing `apps/web` dark palette is the source of truth. shadcn CSS variables must
be initialized to match this palette. After `npx shadcn@latest init`, override the
generated CSS variables in `apps/web/src/styles.css` to align with the values below.

| Role | Value | Usage | Source |
|------|-------|-------|--------|
| Dominant (60%) | `#101214` | Page background, terminal background | styles.css `:root background` |
| Secondary (30%) | `#171a1f` | Auth card surface, header background, form container | styles.css `header`, `.session-card` |
| Tertiary surface | `#0c0e10` | Input background, scrollport, terminal shell | styles.css `input`, `.scrollport` |
| Border | `#2b3137` | Card borders, dividers, header border | styles.css `border-bottom` |

## Interaction Contracts

### Form Validation Timing

- Validate on submit, not on blur, for all auth forms.
- After first submit attempt with errors, switch to on-change validation for already-touched fields.
- Use zod schemas with react-hook-form (shadcn Form pattern). Source: RESEARCH.md Pattern 4.

### Loading State

- Disable the CTA button and replace button text with loading copy while the API request is in flight.
- Do not show a spinner element; text replacement is sufficient for these short-lived requests.

### Error Display

- Display form-level errors (wrong password, server error) as a text message between the last field and the CTA button using shadcn `FormMessage` or an equivalent `<p>` with destructive color (`#e05353`).
- Display field-level errors (invalid email format, password too short) inline below each field using `FormMessage`.
- Never show raw API error codes or stack traces to the user.

### Token Storage and Auth State

- On successful login: `localStorage.setItem('tether_access_token', ...)` and `localStorage.setItem('tether_refresh_token', ...)`.
- On page load: check `localStorage` for a token; if absent, redirect to `/login`.
- On 401 response from any API call: clear tokens, redirect to `/login` with the session-expired message.
- Source: CONTEXT.md D-08, D-09, D-10.

### Password Input

- Use `type="password"` — no show/hide toggle in Phase 5 (deferred complexity).

### Navigation Guard

- The authenticated session shell (`/`) checks for a valid token. If no token exists, redirect to `/login` immediately, without flash of the session list.
