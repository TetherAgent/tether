<objective>
Wire the local Gateway side of the auth model: CLI login and token persistence, direct Gateway HTTP authorization, revoke handling, and short-lived scoped WS tickets. After this plan, direct Gateway access should no longer be LAN-trust-based.
</objective>

<threat_model>
## Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| `~/.tether/auth.json` is world-readable or written non-atomically during refresh | high | Write with `0o600`, replace atomically, and test relogin/refresh behavior. |
| Gateway session write endpoints remain effectively unauthenticated | high | Every write endpoint and `/api/ws-ticket` must validate `normal_client_access` and owner scope before acting. |
| WS tickets remain scope-less UUIDs and can be replayed across sessions/accounts | high | Replace them with short-lived signed tickets containing `accountId`, `workspaceId`, `gatewayId`, `sessionId`, `mode`, `userId`, `deviceId`, and `jti`. |
</threat_model>

## Must Haves

### Truths
- Gateway remains the PTY/session owner, but Phase 5 removes LAN-trust-only writes.
- Management tokens never directly control terminal sessions.
- `auth.json` is a local machine secret and must be treated accordingly.

### Checks
- CLI login exists and persists Gateway auth safely.
- Gateway write routes reject missing/wrong-scope tokens.
- WS tickets are signed, single-use, and scoped.
</must_haves>
