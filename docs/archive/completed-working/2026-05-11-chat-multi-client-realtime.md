# Chat 多端实时同步方案

本文记录 PC 和手机同时打开同一个 chat session 时的当前问题、目标边界、改动范围、TODO 和验收项。

## 结论

当前实现不完整支持多端同时实时观看同一个 chat session。

现状更接近：

```text
单个活跃 client 实时接收 chat 输出
```

目标应该改成：

```text
多个 client 可以同时订阅同一个 chat session，并实时看到同一轮输出。
同一个 chat session 同一时间只允许一轮发送中的请求。
```

也就是说：

- PC 和手机同时打开同一个 chat，可以同时看到 `agent.delta` / `agent.result`。
- 但 PC 和手机不能同时对同一个 session 发两条消息。
- 第二个发送请求应被拒绝，并返回明确错误，例如 `chat_in_progress`。

## 当前问题

### 1. 实时输出按单个 clientId 点发

Relay 收到 `agent.delta` / `agent.result` 后，目前只发给 event payload 里的 `clientId`。

这会导致：

- PC 和手机都打开同一个 session。
- 谁最后绑定，谁更可能收到实时输出。
- 另一端只能等最终消息写入 `gateway_chat_messages` 后刷新历史。
- 如果只有 delta、还没 result，另一端完全看不到中间过程。

### 2. Gateway 侧 chatClientBindings 是单值绑定

Gateway 侧当前是：

```text
sessionId -> clientId
```

后来的订阅或发送会覆盖前一个 client。

这不适合多端同时观看。多端场景里，`clientId` 只能表示“谁发起了这一轮请求”，不能表示“只有谁能看到输出”。

### 3. 多端同时发送会打乱同一个 agent session

如果 PC 和手机同时对同一个 session 发送消息，可能出现：

- 同一个 session 启动两轮 provider 子进程。
- 两轮输出交错。
- `agentSessionId` / usage / result 归属混乱。
- 最终写入的 chat history 顺序不可控。

所以只做广播还不够，必须同时加 session 级 in-flight 锁。

### 4. runner stderr 不等于当前回复失败

实测 Codex / ChatGPT runtime 可能在同一轮回复中先吐出 stderr 诊断事件，例如：

```text
chat_runner_stderr: codex_core::tools::router: error=apply_patch ...
chat_runner_stderr: rmcp::transport::worker: worker quit ...
```

但随后仍然继续收到 `agent.delta`。这说明 `chat_runner_stderr` 不能被前端当作
当前 assistant 回复的终止失败。

前端渲染规则应为：

- 收到 `agent.delta`：继续追加到当前 assistant bubble。
- 收到 `chat_runner_stderr`：只记录为 runtime warning / raw event，不显示 `Reply lost`。
- 收到 `agent.result` / `agent.done`：标记当前回复完成。
- 只有明确 fatal / disconnect / timeout，且当前 turn 没有收到任何 `agent.delta` 或
  `agent.result`，才显示 `Reply lost`。
- 如果当前 turn 已经收到部分 `agent.delta`，后续失败也应保留已有内容，并标记为
  “回复中断”或等价状态，而不是整条丢失。

## 设计方向

### A. 输出按 session 广播

`agent.delta` / `agent.result` / `agent.tool` / `session.error` / `agent.permission_request`
不应该只发给单个 `clientId`。

Relay 应按 `sessionId` 找到所有订阅该 session 的 client，并广播给这些 client。

要求：

- 仍然必须校验账号和 session 归属。
- 只发给当前能访问该 session 的 client。
- `clientId` 可以继续保留在 raw event 里，用来标记发起方，但不能作为唯一投递目标。

### B. Gateway 不再用单值 chatClientBindings 决定观看者

Gateway 侧可以保留发起方 `clientId`，用于错误回包、权限请求等需要定位发起方的场景。

但实时输出是否给哪些客户端看，应交给 Relay 根据 subscriptions 广播。

### C. 同 session 增加 in-flight 锁

在 Gateway 的 chat runner 层增加 session 级运行状态：

```text
sessionId -> active run
```

当某个 session 正在生成回复时：

- 再收到同 session 的 `client.chat`，直接返回错误。
- 错误码建议：`chat_in_progress`。
- 错误消息建议：`当前会话正在回复中`。

释放时机：

- `agent.result` 完成后释放。
- `session.error` / provider 子进程失败后释放。
- 子进程异常退出后释放。
- 后续可加超时兜底，第一版可先用现有 active subprocess 生命周期。

## 暂不做

- 暂不做消息排队。
- 暂不允许同一个 session 并行多轮 agent run。
- 暂不做多端编辑同一草稿输入框同步。
- 暂不做“谁正在输入”的 presence。
- 暂不改变 `gateway_chat_messages` 的历史读取方式。

## 和 raw events 方案的关系

本方案解决“多端能不能同时实时看到同一轮输出”。

`2026-05-11-chat-runtime-raw-events.md` 解决“delta / result / tool / error 是否入库，以及 rawjson 怎么保存”。

两者可以独立实施，但有交叉收益：

- 多端广播保证在线端实时一致。
- raw events 入库保证断线后可以追查和恢复过程。
- 如果后续要做断线后恢复未完成流式输出，两份方案都需要。
